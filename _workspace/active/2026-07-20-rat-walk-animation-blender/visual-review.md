# 쥐 걷기 애니메이션 Blender 시험 제작 비주얼 검토

## 비주얼 판단

이번 작업은 승인된 저폴리 3D 원본 기반 8방향 스프라이트의 **걷기 시험 자산 제작**이다. `rat-host-directional-trial-v1`의 실루엣·제한 팔레트·직교 쿼터뷰 기준을 유지하고, 3D 게임플레이 루트와 Unity 통합은 건드리지 않는다. 64 px 셀에서는 다리의 세밀한 관절보다 머리·꼬리·몸통의 리듬과 접지가 읽히는 것이 우선이다.

## 추천 제작 기준

### 1. 루프 길이와 위상

- 루프는 **8프레임**, Blender 타임라인 `1`~`8`을 사용한다. `9`는 프레임 `1`의 반복 재생 결과일 뿐 별도 PNG로 출력하지 않는다.
- 모든 방향은 같은 프레임 번호와 같은 보행 위상으로 출력한다. 방향을 바꾸어도 재생 위상이 초기화된 것처럼 보이지 않는 기준이다.
- 위상은 `1` 좌전/우후 접지, `3` 통과, `5` 우전/좌후 접지, `7` 통과로 둔다. `2·4·6·8`은 인접 키포즈의 명확한 중간 포즈로 둔다.
- 뒷다리와 앞다리는 대각선 쌍으로 교대한다. 꼬리는 몸통 회전의 반대쪽으로 작게 지연시켜 균형감을 주되, 큰 좌우 휘두름으로 캔버스 외곽이나 실루엣이 매 프레임 변하지 않게 한다.
- 64 px 시험 출력에서는 8프레임이 충분한 최소 단위다. 프레임 수와 실제 재생 속도(FPS)는 Unity 반입/플레이 검증 뒤 별도 승인으로 확정한다.

### 2. 8방향 카메라·조명·출력 기준

정지 시험 자산의 아래 값을 변경하지 않는다.

- 방향 순서: `S, SW, W, NW, N, NE, E, SE` (`index 0`~`7`)
- source yaw: `-53.130102, -25.904474, 36.869898, 99.644270, 126.869898, 154.095526, -143.130102, -80.355730`
- 투영과 카메라: 직교, `camera_to_object [6.0, -8.0, 6.0]`, `scale_px_per_unit 9.0`
- 조명: 단일 key light `[-0.45, -0.55, 1.0]`; 팔레트·명암 단계와 투명 배경도 정지 시험과 동일
- 각 PNG 캔버스: `64 × 64 px`; 공통 피벗: 상단 좌표 기준 `(32, 48)`; 안티앨리어싱 없음, Point/Nearest 확대 전제

프레임마다 카메라, 조명, 직교 스케일, 오브젝트의 root transform을 바꾸지 않는다. 방향은 한 원본을 yaw 회전해 얻으며, 프레임 `f`는 8방향 모두 `f`의 포즈여야 한다. 프레임 번호는 2자리 1기준(`f01`~`f08`)으로 통일한다.

### 3. 실제 발바닥·pivot·수직 튐

- 원본의 축은 기존 OBJ와 같이 `+X forward`, `+Z up`, root pivot `(0, 0, 0)`을 유지한다. 애니메이션은 다리/꼬리 등 자식 요소에 주고 root의 위치·스케일·Z 이동은 키를 만들지 않는다.
- 지면은 원본 로컬 `Z=0`이다. 접지 위상의 각 발바닥 최저점은 이 평면에 닿고 관통하지 않아야 한다. 통과 위상에서도 지지 다리 최소 하나는 `Z=0`을 유지한다.
- 전 프레임·전 방향에서 발바닥 기준점과 스프라이트 pivot의 수직 기준은 정지 포즈와 같아야 한다. 렌더 결과에서 pivot y=48을 기준으로 바닥선이 상하로 바뀌면 불합격이다.
- 몸통의 보행 리듬은 다리 관절의 상대 움직임으로 표현한다. 64 px에서 1 px 이상의 몸통/root 수직 왕복은 금지하고, 전체 실루엣이 들리거나 지면을 뚫는 프레임은 수정한다.
- 검토 시 각 방향의 `f01 → f08 → f01` 반복과 대각선 방향을 우선 확인한다. 발바닥 바닥선, root 위치, 화면 점유 크기, 꼬리/귀 잘림을 프레임별로 대조한다.

### 4. 원본과 렌더 출력 위치·비덮어쓰기 경계

기존 완료 패킷 `_workspace/completed/2026-07-16-2026-07-16-rat-8-direction-trial-asset/artifacts/` 및 그 안의 `rat-00-s.png`~`rat-07-se.png`, `rat-8dir-sheet.png`, `source/rat-trial-v1.obj`는 읽기 전용 기준 자산으로 취급한다.

이번 시험의 권장 산출 위치는 아래와 같다.

```text
_workspace/active/2026-07-20-rat-walk-animation-blender/
  artifacts/
    source/
      rat-walk-trial-v1.blend
      rat-walk-trial-v1.obj             # 필요할 때만, 기존 OBJ를 덮어쓰지 않는 별도 export
    renders/
      rat-walk-f01-00-s.png
      ...
      rat-walk-f08-07-se.png
      rat-walk-8dir-f01-sheet.png       # 선택: 프레임별 방향 시트
      ...
      rat-walk-8dir-f08-sheet.png
    walk-frame-map.csv
    render-settings-walk.json
```

`walk-frame-map.csv`에는 `frame, direction_index, direction, source_yaw_degrees, file, phase`를, `render-settings-walk.json`에는 위 고정 카메라·조명·캔버스·피벗과 Blender 버전을 기록한다. 출력명과 폴더가 다르면 정지 PNG 및 Unity의 기존 자산을 덮어쓸 수 없다.

### 5. 이번 작업에서 제외하는 Unity 통합 경계

이번 시험은 Blender 원본과 렌더 PNG의 제작·검토까지만 포함한다. 다음은 명시적으로 제외한다.

- `UnityProject/`의 씬, 코드, 프리팹, Sprite Import 설정, PPU, AnimationClip/Animator, URP·ProjectSettings 변경
- 기존 정지 스프라이트의 교체·이동·삭제 및 걷기 프레임의 Unity 반입
- 이동 속도와 프레임 재생 속도의 연결, 방향 전환 시 위상 유지 로직, 콜라이더·위험 판정·상호작용 위치 변경
- 런타임 접지, 그림자, 가림, 실제 플레이 검증의 완료 주장

렌더 출력이 수용된 뒤 Unity 통합을 별도 승인·작업 패킷으로 열고, 그때 8방향 전환, 피벗 흔들림, 발 미끄러짐, 3D 환경과의 가림·접지·그림자, 게임플레이 판정 보존을 검증한다.

## 플레이스홀더 기준

- 본 자산은 `trial-placeholder-not-final-art` 상태를 유지한다. 최종 털 표현, 고해상도 셰이딩, 새로운 색상 확장은 넣지 않는다.
- 저해상도에서 판독되지 않는 다리 세부 묘사는 추가하지 않는다. 필요하면 단순한 굵기·명암 차이로 앞/뒤 다리의 교대를 읽히게 한다.

## 검증 방법

1. Blender에서 root transform과 카메라·조명이 모든 프레임에서 고정인지 확인한다.
2. 8방향 × 8프레임 총 64 PNG의 크기, 투명 배경, 파일명, frame/direction 매핑을 확인한다.
3. 각 방향을 반복 재생해 `f08 → f01` 이음새, 바닥선, pivot, 귀·꼬리 clipping, 1 px 이상 수직 튐을 확인한다.
4. 정지 시험 PNG의 해시와 위치가 변하지 않았고 Unity 프로젝트가 변경되지 않았는지 확인한다.

## 승인 필요

- 사용자 승인으로 Blender 시험 제작과 신규 걷기 프레임 생성은 진행 가능하다.
- Unity 반입, 걷기 클립 재생 속도·프레임 수의 제품 확정, 정지 자산 교체는 별도 사용자 승인 대상이다.
