# 작업 패킷: 내부 미니게임 유형 확장

## 목표

기존 내부 바이러스 미니게임을 백혈구 회피 하나로 고정하지 않기 위해, 다음 유형 후보로 `면역 신호 억제` 리듬형 미니게임을 설계한다.

## 선택한 유형

- 유형명: `면역 신호 억제`
- 플레이 감각: 리듬 입력
- 기획 의미: 면역계의 경보 신호가 박자처럼 퍼지고, 바이러스가 타이밍에 맞춰 그 신호를 끊는다.

## 범위

- 설계 문서와 수용 기준 정리
- 기존 `신호 억제` 후보를 리듬형으로 구체화
- 발생 조건, 성공/실패, 보상/디메리트, 난이도 기준 정의
- 다음 구현 작업 단위 후보 정리

## 비범위

- Unity 코드 구현
- 새 씬/프리팹/UI 생성
- 기존 백혈구 회피 미니게임 대체
- 여러 미니게임 무작위 선택 시스템 구현
- 음악/사운드 에셋 제작
- 최종 밸런스 수치 확정

## 담당 배정

| 담당 | 역할 | 산출물 |
| --- | --- | --- |
| 프로젝트 조정 에이전트 | 작업 범위와 기록 관리 | 작업 패킷, 작업 로그 |
| 게임플레이 루프 에이전트 | 유형 규칙과 수용 기준 설계 | 설계 문서 반영 |
| 기획 정리 에이전트 | 기존 내부 미니게임 방향과 충돌 확인 | 범위/문서 충돌 검토 |
| 문서/릴리즈 에이전트 | 문서 최신화 | `docs/design/encounters/` 변경 |
| 프로젝트 총괄 관리자 에이전트 | 승인 게이트와 다음 구현 범위 판정 | 내부 판정 기록 |

## 승인 게이트 판단

- 이번 작업은 설계/문서 확장으로 한정한다.
- 새 내부 미니게임 구현은 승인 게이트에 해당하므로 이번 작업에서 실행하지 않는다.
- 구현으로 넘어가려면 별도 구현 계획과 사용자 승인이 필요하다.

## 수용 기준

1. `면역 신호 억제`가 내부 미니게임 유형 후보로 문서화된다.
2. 이 유형이 리듬 입력 기반이라는 점이 명확해야 한다.
3. 기존 백혈구 회피 미니게임을 대체하지 않고, 두 번째 유형 후보로 유지해야 한다.
4. 발생 원인, 성공 조건, 실패 조건, 보상, 디메리트가 정리되어야 한다.
5. 1차 프로토타입 구현 범위가 여러 미니게임으로 즉시 확대되지 않도록 구현 보류를 명시해야 한다.
6. 다음 구현 시 최소 프로토타입으로 검증할 수 있는 작업 단위가 남아야 한다.
