## 2. 계획 초안 수용 기준 커버리지 검토

결론: 계획 초안은 핵심 루프의 큰 흐름을 대부분 덮고 있으나, 구현 완료 조건으로 쓰려면 일부 수용 기준을 더 구체화해야 한다. QA 관점에서는 `조건부 수용`이다.

| 핵심 수용 기준 | 초안 커버리지 | QA 판단 |
| --- | --- | --- |
| 쥐가 의도대로 이동한다 | `WASD` 이동, 방향 전환, 충돌 확인 시나리오가 있다 | 커버됨. 단, 카메라 기준 이동인지 월드 기준 이동인지 최종 결정이 필요하다. |
| 면역 경계도가 오르는 이유가 보인다 | 시간 상승, 위험 요소 접촉 상승, 경고 색상 또는 피드백이 있다 | 대체로 커버됨. 위험 요소가 왜 위험한지 화면 피드백 기준을 더 명확히 해야 한다. |
| 면역 경계도 100%에서 내부 미니게임으로 전환된다 | 100% 이벤트, 모드 루트 활성/비활성 전환, 중복 전환 확인이 있다 | 커버됨. 100% 도달 이벤트가 한 번만 발생하는지 자동 테스트 후보로 분리해야 한다. |
| 미니게임 목표가 이해된다 | 변이 조각 `0/3` UI, 백혈구 회피, 안정도 표시가 있다 | 커버됨. 시작 직후 목표 표시와 실패 조건 표시 기준을 추가하는 편이 좋다. |
| 변이 선택 후 쥐 모드에 변화가 적용된다 | 세 변이 효과와 변이별 검증 시나리오가 있다 | 부분 커버. `포유류 적응`의 임시 효과가 아직 승인 대기라 완료 기준으로 확정할 수 없다. |
| 실패 루프가 검증 가능하다 | 안정도 0 실패, 재시도 패널, 재시도 시나리오가 있다 | 커버됨. 재시도 시 초기화 범위가 필요하다. |
| UI가 플레이를 방해하지 않는다 | 16:9 겹침 확인, 긴 텍스트 확인이 있다 | 부분 커버. 확인 해상도와 각 모드별 필수 화면 상태를 고정해야 한다. |
| 카메라가 플레이 공간을 제대로 보여준다 | 고정 쿼터뷰 전제가 있다 | 부족. 쥐 모드와 바이러스 모드 각각의 프레이밍 기준이 필요하다. |
| 구현 범위가 확장되지 않는다 | 제외 범위와 범위 초과 감지 기준이 있다 | 커버됨. 추가 적 타입, 다중 숙주, 최종 아트는 별도 승인으로 분리되어 있다. |
| 완료 주장을 검증으로 증명한다 | 통합 검증 계획과 QA 요청이 있다 | 부분 커버. 컴파일, PlayMode, 빌드 중 필수 게이트를 명시해야 한다. |

구현 전 보완이 필요한 기준:

- `포유류 적응`의 1차 효과를 `특정 통로 접근` 또는 `보상 지점 접근` 중 하나로 확정한다.
- 실패 후 재시도 시 초기화할 값과 유지할 값을 명시한다.
- 성공과 실패가 같은 프레임에 발생할 때 우선순위를 정한다.
- `RatHostPrototype.unity`를 빌드 설정에 포함할지 결정한다.
- 입력 구현이 Input System 기반인지 기존 `UnityEngine.Input` 기반인지 결정한다.
- UI/카메라 확인 해상도와 캡처 기준을 명시한다.

## 3. 수동 플레이 체크리스트

아래 체크리스트는 구현 후 사용자 승인된 범위에서 실행할 항목이다. 현재는 실행하지 않았다.

### 사전 확인

- `RatHostPrototype` 씬이 의도한 작업 씬으로 열려 있다.
- 콘솔에 컴파일 오류가 없다.
- 시작 시 활성 모드는 쥐 숙주 모드다.
- 쥐 숙주 모드 HUD에는 숙주 생명력, 면역 경계도, 현재 모드 또는 현재 변이 정보가 보인다.
- 바이러스 미니게임 오브젝트와 UI는 시작 시 보이지 않거나 입력을 받지 않는다.

### 성공 루프

1. `WASD`로 쥐를 30초 이상 이동한다.
2. 쥐가 이동 방향을 향하는지 확인한다.
3. 벽, 막힌 배관, 맵 경계에서 의도하지 않은 통과가 없는지 확인한다.
4. 관심 지점 앞에서 `Space`를 눌러 상호작용 1종이 동작하는지 확인한다.
5. 위험 요소에 닿지 않은 상태로 면역 경계도가 시간에 따라 상승하는지 확인한다.
6. 위험 요소에 닿았을 때 면역 경계도 상승 속도 또는 피드백이 달라지는지 확인한다.
7. 면역 경계도 100%에서 쥐 모드 입력이 멈추고 내부 바이러스 미니게임으로 전환되는지 확인한다.
8. 바이러스 모드에서 `WASD`로 바이러스가 아레나 안에서 움직이는지 확인한다.
9. 백혈구 접촉 시 바이러스 안정도가 감소하는지 확인한다.
10. 변이 조각 3개를 수집하면 변이 선택 화면이 표시되는지 확인한다.
11. 변이 1개를 선택하면 쥐 모드로 복귀하는지 확인한다.
12. 복귀 후 선택한 변이 효과가 쥐 모드에서 확인되는지 확인한다.

### 실패 루프

1. 내부 바이러스 미니게임에 진입한다.
2. 백혈구와 반복 접촉해 바이러스 안정도를 0까지 낮춘다.
3. 실패 패널 또는 재시도 UI가 표시되는지 확인한다.
4. 실패 상태에서 쥐 모드 입력이나 변이 선택 입력이 잘못 동작하지 않는지 확인한다.
5. 재시도 입력을 실행한다.
6. 바이러스 안정도, 변이 조각 수, 백혈구 위치, 수집된 조각 상태가 의도한 초기값으로 돌아왔는지 확인한다.
7. 재시도 후 조각 3개 수집 성공 루프가 다시 가능한지 확인한다.

### 변이별 확인

- `잠복 강화`: 같은 조건에서 30초 동안 면역 경계도 상승량이 선택 전보다 낮아지는지 비교한다.
- `신경 조종`: 같은 구간 이동 시간 또는 같은 상호작용 완료 시간이 선택 전보다 짧아지는지 비교한다.
- `포유류 적응`: 승인된 효과에 따라 특정 통로 또는 보상 지점 접근이 선택 전에는 불가하고 선택 후에는 가능한지 확인한다.
- 같은 성공 이벤트에서 변이를 두 번 선택할 수 없는지 확인한다.
- 변이 선택 후 현재 변이 표시가 쥐 모드 HUD에 반영되는지 확인한다.

### 부정/경계 확인

- 면역 경계도가 100%를 초과 표시하지 않는지 확인한다.
- 바이러스 안정도가 0 아래로 표시되지 않는지 확인한다.
- 변이 조각 수가 `3/3`을 초과 표시하지 않는지 확인한다.
- 모드 전환 중 `Space` 또는 이동 입력을 연타해도 중복 전환이 발생하지 않는지 확인한다.
- 변이 선택 패널 또는 재시도 패널이 열린 동안 뒤쪽 플레이 오브젝트가 잘못 입력을 받지 않는지 확인한다.

## 4. EditMode/PlayMode 테스트 권장 범위

### EditMode 권장

순수 로직으로 분리 가능한 항목은 EditMode 테스트를 우선 권장한다.

- `ImmuneAlertMeter`
  - 시간 경과 상승
  - 위험 요소 추가 상승
  - `잠복 강화` 적용 시 상승량 감소
  - 0~100 범위 제한
  - 100% 도달 이벤트 1회 발행
- `PrototypeGameState`
  - 현재 모드 변경
  - 선택 변이 저장
  - 재시도와 복귀 시 상태 초기화 정책
- `VirusMinigameController`
  - 변이 조각 수 카운트
  - 바이러스 안정도 감소
  - 조각 3개 수집 성공 이벤트
  - 안정도 0 실패 이벤트
  - 성공/실패 동시 조건 우선순위
- `MutationEffectApplier`
  - 잠복 강화가 면역 상승 배율에 반영된다.
  - 신경 조종이 이동 또는 상호작용 수치에 반영된다.
  - 포유류 적응이 승인된 접근 조건에 반영된다.

### PlayMode 권장

GameObject 활성 상태, UI, 씬 연결, 입력 흐름은 PlayMode 테스트 또는 수동 플레이로 확인한다.

- 씬 시작 시 `RatHostMode`만 활성이고 `VirusMinigameMode`, `MutationChoicePanel`, `RetryPanel`은 비활성이다.
- 면역 경계도 100% 조건을 주입하면 바이러스 미니게임 루트와 해당 HUD가 활성화된다.
- 미니게임 성공 이벤트를 주입하면 변이 선택 화면이 표시된다.
- 변이 선택 후 쥐 모드 루트와 쥐 HUD가 다시 활성화된다.
- 미니게임 실패 이벤트를 주입하면 재시도 패널이 표시된다.
- 재시도 입력 후 미니게임 상태가 초기화된다.
- 모드 전환 후 이전 모드의 입력, 충돌, UI가 남아 동작하지 않는다.

권장 최소 자동화 기준:

- EditMode 테스트는 핵심 수치/상태 로직에 대해 필수로 둔다.
- PlayMode 테스트는 최소한 모드 전환 happy path와 실패 재시도 흐름을 포함한다.
- 자동화가 어렵거나 비용이 큰 조작감, 카메라, UI 겹침은 수동 체크리스트로 보완한다.

## 5. 컴파일/플레이/빌드 완료 조건 제안

구현 완료 주장은 단계별로 나누는 것이 안전하다.

### 내부 구현 완료 조건

- 사용자 승인된 범위 안에서만 Unity 씬, 코드, 에셋이 변경되어 있다.
- Unity 에디터 컴파일 오류가 없다.
- 신규 또는 수정된 EditMode 테스트가 통과한다.
- 최소 PlayMode 테스트가 통과하거나, PlayMode 자동화 제외 사유가 기록되어 있다.
- 수동 플레이 체크리스트의 성공 루프와 실패 루프가 통과한다.
- 세 가지 변이 효과가 각각 한 번 이상 확인되어 있다.
- 콘솔에 루프 진행을 막는 예외가 없다.
- 실행하지 않은 빌드나 테스트를 통과로 기록하지 않는다.

### 플레이어블 프로토타입 완료 조건

내부 구현 완료 조건에 더해 다음을 요구한다.

- Windows PC 대상 빌드가 성공한다.
- 빌드 실행본에서 성공 루프 1회와 실패 루프 1회를 확인한다.
- 빌드 실행본에서 UI 겹침과 카메라 프레이밍 기준을 통과한다.

### 이번 QA 계획 검토 완료 조건

- `qa-review.md`가 작성되어 있다.
- 미실행 검증과 남은 위험이 분리되어 있다.
- 구현 후 검증 실행 범위가 사용자 승인 대상으로 남아 있다.

## 6. UI 겹침과 카메라 프레이밍 확인 기준

### UI 확인 화면 크기

PC 우선 프로토타입 기준으로 다음 해상도를 권장한다.

- `1920x1080`: 기본 검증 해상도
- `1600x900`: 중간 창 크기
- `1366x768`: 낮은 노트북 해상도
- `1280x720`: 최소 16:9 기준
- 필요 시 `2560x1440`: 고해상도 확장 확인

모바일 해상도와 세로 화면은 1차 프로토타입 완료 기준에 포함하지 않는다.

### UI 통과 기준

- 숙주 생명력, 면역 경계도, 현재 변이 표시가 서로 겹치지 않는다.
- 바이러스 안정도와 변이 조각 수가 서로 겹치지 않는다.
- 변이 선택 화면의 세 선택지가 모두 클릭 또는 키보드 선택 가능하다.
- 한국어 변이 이름과 효과 설명이 버튼 또는 패널 밖으로 넘치지 않는다.
- 실패/재시도 패널이 HUD와 의미 없이 겹치지 않는다.
- 모드 전환 시 이전 모드 HUD가 남아 새 모드 HUD와 겹치지 않는다.
- UI가 쥐, 위험 요소, 변이 조각, 백혈구의 핵심 위치를 가리지 않는다.

### 카메라 프레이밍 통과 기준

쥐 숙주 모드:

- 쥐가 기본 이동 중 화면 중심 또는 의도한 추적 위치에서 벗어나지 않는다.
- 하수도 주요 경로, 위험 요소, 관심 지점, 포유류 적응 접근 지점이 식별 가능하다.
- 벽이나 장식 오브젝트가 쥐를 장시간 가리지 않는다.
- 고정 쿼터뷰에서 이동 방향을 판단할 수 있다.

바이러스 미니게임 모드:

- 아레나 경계가 화면 안에서 이해 가능하다.
- 바이러스, 백혈구, 변이 조각이 동시에 식별 가능하다.
- 백혈구 접촉 위험을 피할 수 있을 만큼 주변 공간이 보인다.
- 저해상도 렌더링 또는 후처리 때문에 UI 텍스트가 흐려지지 않는다.

## 8. 남은 위험

- 승인 패킷 전체 승인 여부가 아직 확인되지 않았으므로 구현 시작 조건이 열려 있다.
- `포유류 적응`의 임시 효과가 확정되지 않아 변이별 완료 기준 하나가 미정이다.
- `RatHostPrototype.unity`를 Build Settings에 추가할지 미정이라 빌드 검증 기준이 흔들릴 수 있다.
- Input System 사용 여부가 확정되지 않으면 입력 테스트와 구현 책임이 달라진다.
- 단일 씬 루트 활성/비활성 방식은 빠르게 구현하기 좋지만, 모드 전환 시 이전 모드 상태나 입력이 남을 위험이 있다.
- 실패 후 재시도 초기화 범위가 불명확하면 같은 버그를 수동 플레이에서만 발견할 수 있다.
- 면역 경계도 상승 수치가 과하면 탐험이 끊기고, 너무 낮으면 내부 미니게임 전환 이유가 약해질 수 있다.
- 위험 요소 피드백이 약하면 플레이어가 왜 면역 경계도가 오르는지 이해하지 못할 수 있다.
- UI와 저해상도 렌더링을 함께 적용하면 도트풍 3D 의도와 UI 가독성이 충돌할 수 있다.
- 현재 베이스라인상 활성 씬과 빌드 설정 씬 기준이 다르므로 작업 씬 정책을 먼저 정해야 한다.

## 9. 완료 판단 기준

계획 초안의 QA 기준은 다음 조건을 만족하면 수용 가능하다.

- 핵심 루프 여섯 단계가 각각 수동 플레이 체크 항목으로 검증 가능하다.
- 실패 루프와 재시도 루프가 성공 루프와 별도로 검증 가능하다.
- 세 가지 변이 효과가 각각 관찰 가능한 결과로 정의되어 있다.
- 컴파일, 자동 테스트, 수동 플레이, 빌드 중 어떤 항목이 완료 게이트인지 명시되어 있다.
- 미실행 검증을 통과로 표현하지 않는다.
- 구현 범위 밖 항목이 별도 승인 대상으로 분리되어 있다.

현재 계획 초안에 대한 QA 판단:

- 핵심 루프와 주요 수동 검증 시나리오는 충분히 잡혀 있다.
- 자동 테스트 범위, 빌드 완료 조건, UI/카메라 기준은 이 문서의 제안을 반영하면 완료 기준으로 사용할 수 있다.
- `포유류 적응` 효과, 재시도 초기화 범위, 입력 방식, 빌드 설정 씬 정책은 구현 전 확정이 필요하다.
