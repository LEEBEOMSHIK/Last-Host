# 작업 패킷: 면역 신호 억제 접근 예고 확장

## 목적

`면역 신호 억제` 미니게임에서 신호가 판정선에 가까워지는 순간을 더 명확히 보여준다. 사운드, 새 노트 프리팹, 신규 아트 에셋 없이 기존 런타임 HUD와 판정 모델 상태만 확장한다.

## 범위

- 신호 접근 예고 시간 설정 추가
- 신호 접근 예고 활성 여부와 강도 상태 노출
- HUD 신호 마커/판정선/정확 구간 색상과 크기 피드백 강화
- EditMode 테스트 추가
- 문서에 이번 후속 확장 완료 범위 기록

## 제외 범위

- 사운드 싱크 또는 오디오 에셋 추가
- 전용 노트 프리팹/이펙트 에셋 생성
- 기본 내부 미니게임 타입 변경
- 백혈구 회피 난이도 수치 변경
- 씬 배치, ProjectSettings, 패키지 변경

## 수용 기준

1. 면역 신호 억제 진입 직후에는 기존처럼 다음 신호까지 남은 시간이 표시된다.
2. 신호가 정확 판정 구간에 가까워지면 `신호 접근` 상태와 cue intensity가 노출된다.
3. 정확 판정 구간에서는 기존 `지금 차단` 상태와 성공 판정이 유지된다.
4. HUD는 접근 예고 상태에서 마커/판정선/정확 구간 색상 또는 크기 변화를 표시한다.
5. 기존 백혈구 회피 미니게임과 미커밋 백혈구 수정 작업을 되돌리지 않는다.
6. EditMode 테스트와 가능한 Unity MCP Play 체크로 검증한다.

## 2026-07-16 검증 종결 재개

### 사용자 승인과 담당

- 사용자 승인: 2026-07-16 `면역 신호 억제 접근 예고 검증 종결` 진행 승인.
- 현재 단계: 기존 구현을 변경하지 않는 검증 종결 재개.
- 담당 에이전트: QA/검증 에이전트.
- 후속 게이트: QA 완료 판단 뒤 프로젝트 총괄 관리자 에이전트가 내부 승인 가능 여부를 판정한다.

### 재개 범위

- Unity Test Runner에서 `LastHost.Prototype.Tests` EditMode 테스트를 정식 실행하고 결과를 기록한다.
- Unity MCP로 `RatHostPrototype` 씬의 Play 진입과 종료를 각각 확인한다.
- Play 상태에서 다음 두 경로 중 실제로 실행한 경로를 명시적으로 분류한다.
  - `실제 키 입력 검증`: Game View 포커스 증적을 확보하고 네이티브 F6 입력으로 면역 신호 억제 상태에 진입한다.
  - `MCP 직접 상태 전환 대체 검증`: 실제 키 입력이 불가능하면 승인된 세션 API 또는 MCP 명령으로 상태를 직접 전환한다. 이 결과를 F6 입력 수신 증명으로 사용하지 않는다.
- 실행한 경로에서 `신호 접근` 예고, cue intensity, HUD 마커/판정선/정확 구간 변화와 `지금 차단` 상태를 확인한다.
- Unity Console Error/Warning, Play 종료 성공 여부, 활성 씬 dirty 상태를 확인한다.

### 금지 범위

- Unity 코드, 씬, 테스트, ProjectSettings, 패키지, 에셋을 변경하지 않는다.
- `.codex/config.toml`과 이전 완료 작업 패킷을 수정하거나 이동하지 않는다.
- 일반 `SendKeys`, 직접 상태 전환 명령 반환, 컴파일 스모크만으로 실제 F6 키 입력 검증 통과를 선언하지 않는다.
- QA 완료 판단과 프로젝트 총괄 관리자 판정 전에는 작업 완료 또는 완료 보관을 주장하지 않는다.

### 검증 종결 완료 기준

1. EditMode 정식 테스트의 명령, 결과 파일 또는 Test Runner 결과, 통과/실패 수가 기록되어 있다.
2. Unity MCP Play 진입과 종료 결과가 모두 기록되어 있다. 실행 불가 시에는 차단 메시지와 재개 조건이 기록되어 있다.
3. 실제 키 입력 또는 직접 상태 전환 중 사용한 경로가 검증 표준에 맞게 분류되어 있다.
4. 접근 예고 상태, HUD 변화, Console Error/Warning, 활성 씬 dirty 상태가 함께 기록되어 있다.
5. 실제 키 입력으로 확인한 항목과 대체 검증으로만 확인한 항목이 분리되어 있다.
6. `verification.md`에 QA/검증 에이전트의 미검증 항목, 남은 위험, 완료 가능 여부와 근거가 남아 있다.
7. QA 완료 가능 판정 뒤 프로젝트 총괄 관리자 판정이 `completion-report.md`에 기록되어야 완료 보관할 수 있다.
