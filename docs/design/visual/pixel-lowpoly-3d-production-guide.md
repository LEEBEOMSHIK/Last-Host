# 도트풍 하이브리드 2.5D 제작 가이드

## 목적과 적용 범위

`마지막 숙주`의 화면은 순수 2D 스프라이트 게임이 아니다. 실제 3D 환경·게임플레이 공간·카메라 위에서 캐릭터를 저폴리 3D 원본으로 제작하고, 쥐 숙주 프로토타입 런타임에서는 이를 렌더한 8방향 스프라이트로 표시하는 도트풍 하이브리드 2.5D다. 이 가이드는 이후 원본 모델·스프라이트·환경·씬·URP 작업이 같은 방향을 유지하도록 돕는다.

아래의 해상도, 폴리곤 밀도, 조명 값은 확정된 Unity 설정이 아닌 **권장 시작 기준**이다. 실제 에셋 제작, Unity Import 변경, URP Renderer·카메라·후처리 변경은 별도 작업 패킷, 사용자 승인, Unity 검증을 거쳐야 한다.

## 1. 표현의 핵심: 3D 공간, 일관된 캐릭터 원본, 읽히는 픽셀 화면

- 공간, 충돌, 환경, 카메라는 3D로 구성한다. 쥐 숙주 탐험은 고정 쿼터뷰를 우선하고, 내부 바이러스 미니게임은 탑다운 또는 고정 쿼터뷰를 우선한다.
- 쥐·바이러스·백혈구는 하나의 승인된 저폴리 3D 원본과 리그를 방향·비율·애니메이션 일관성의 기준으로 둔다. 런타임에는 이 원본을 같은 카메라·조명·화면 점유 크기로 렌더한 8방향 스프라이트를 사용한다.
- 도트감은 캐릭터 스프라이트의 픽셀 격자, 환경의 픽셀풍 표면, 낮은 내부 렌더 해상도 또는 의도적인 픽셀화에서 만든다.
- 한 화면에서 먼저 읽혀야 하는 것은 숙주, 위험 요소, 상호작용 지점, 이동 가능한 경로, 현재 목표다. 미세한 표면 장식보다 실루엣·명도 대비·색상 구분을 먼저 잡는다.
- 저폴리와 저해상도는 단순히 품질을 낮추는 수단이 아니다. 모델의 큰 면, 제한된 팔레트, 안정적인 카메라가 함께 갖춰져야 의도된 스타일로 읽힌다.

## 2. 권장 제작 순서

1. 씬의 목표 해상도와 카메라 구도를 정하고, 회색 또는 제한 팔레트 블록아웃에서 이동·상호작용·위험 가독성을 확인한다.
2. 쥐, 바이러스, 백혈구, 위험 요소, 배관처럼 게임플레이를 구분해야 하는 대상부터 저폴리 실루엣을 만든다.
3. 캐릭터는 저폴리 원본의 리그·팔레트·렌더 카메라·조명 기준을 정하고 8방향 시험 스프라이트를 출력한다. 환경은 공통 텍셀 밀도와 제한 팔레트에 맞춰 픽셀 텍스처와 UV를 적용한다.
4. 캐릭터 스프라이트의 방향 매핑·피벗·월드 크기·가림을 3D 게임플레이 루트에 맞춘 뒤, 낮은 내부 렌더 해상도의 정수 배율 확대를 시험한다.
5. 조명·그림자·포그·색보정을 최소 구성으로 얹고, 프리렌더 명암과 3D 환경 조명이 충돌하지 않는지 확인한다.
6. 목표 해상도와 비목표 해상도 모두에서 정지·이동·빠른 방향 전환 화면을 확인한 뒤, 적용값을 작업 기록에 남긴다.

## 3. 저폴리 모델 기준

### 실루엣과 형태

- 캐릭터는 머리·몸통·다리·꼬리처럼 행동과 방향을 읽게 하는 큰 덩어리를 먼저 만든다. 쥐는 꼬리와 머리 방향, 바이러스는 외곽 돌기, 백혈구는 위협적인 외곽 형태가 먼 거리에서도 구분되어야 한다.
- 환경은 걷는 면, 막힌 면, 위험 면, 상호작용 면을 형태와 색으로 구분한다. 예를 들어 배관의 입구·밸브·위험 누수는 장식보다 먼저 읽혀야 한다.
- 작은 홈, 둥근 모서리, 고밀도 베벨, 미세한 반복 장식은 피한다. 화면 크기에서 보이지 않는 기하학은 텍스처나 색상 블록으로 대체한다.
- 기울어진 긴 면과 아주 얇은 면은 낮은 해상도에서 깜빡이기 쉽다. 큰 면으로 단순화하고 카메라 각도에서 윤곽이 겹치지 않는지 확인한다.

### 밀도와 머티리얼

| 대상 | 권장 시작 밀도 | 머티리얼 기준 |
| --- | --- | --- |
| 쥐·바이러스·백혈구 원본 | 큰 덩어리 중심의 수백~약 1,500 삼각형 범위에서 시작 | 방향별 스프라이트를 재생산하기 위한 원본으로 관리하며 대상당 1~2개 단순 머티리얼 사용 |
| 작은 상호작용 소품 | 12~250 삼각형 정도의 단순 프리미티브·모듈 조합에서 시작 | 공용 팔레트·공용 셰이더를 우선 |
| 배관·벽·바닥 모듈 | 화면에 보이는 윤곽을 보존하는 낮은 밀도, 반복 모듈화 | 넓은 면의 미세한 PBR 효과보다 색상 블록·명도 차를 우선 |

이 수치는 에셋 예산 확정값이 아니다. 화면 점유율, 애니메이션, 카메라 거리, PC 성능을 함께 보고 조정한다. 머티리얼은 우선 불투명하고 단순한 표면을 사용하며, 금속성·높은 매끄러움·복잡한 레이어드 셰이더는 스타일과 가독성을 실제로 개선할 때만 별도 검토한다.

## 4. 캐릭터 스프라이트와 환경 픽셀 텍스처 기준

### 프리렌더드 8방향 캐릭터

- 8개 방향은 런타임 고정 카메라 기준으로 구분하며, 방향 이름·배열 순서·Animator 매핑을 한 일관성 시트에 기록한다.
- 모든 방향과 프레임은 같은 투영, 화면 점유 크기, 캔버스, 피벗, 알파 여백, 조명 기준으로 출력한다.
- 공통 피벗은 지면 접촉점 또는 3D 이동 루트 중심을 기준으로 하고, SpriteRenderer는 3D 게임플레이 루트의 시각 자식으로 둔다.
- 스프라이트 프레임 교체는 콜라이더, 이동 루트, 위험 판정, 상호작용 위치를 바꾸지 않는다.
- 방향 전환 시 애니메이션 위상이 불필요하게 초기화되거나 경계에서 두 방향이 빠르게 왕복하지 않는지 검증한다.
- 정확한 셀 해상도, PPU, 프레임 수, 재생 속도, 8방향 개별 렌더와 좌우 반전 사용 범위는 시험 에셋 후 결정한다.

### 프리렌더 게임플레이 오브젝트 공통 기준 (쥐 v4 시험 기준)

쥐 v4의 `128×128` 출력은, 같은 화면 점유와 더 높은 샘플 밀도를 함께 검증하기 위한 공통 **기본 규격**이다. 이후의 프리렌더 캐릭터와 전경 게임플레이 오브젝트는 아래 기준에서 시작한다. 이 규격은 3D 환경 텍스처, 원거리 배경, HUD에 일괄 적용하는 규칙이 아니다.

| 항목 | 기본 규격 | 의도 |
| --- | --- | --- |
| 출력 | 프레임당 `128×128` RGBA PNG | 같은 캔버스 안에서 64px 시험본보다 실루엣·다리·꼬리의 계단을 더 안정적으로 보존 |
| Unity Sprite | Single, PPU `64` | `128 / 64 = 2` Unity unit으로, 기존 `64 / 32 = 2` Unity unit 캔버스 폭과 월드 표시 크기 보존 |
| 필터·압축 | Point (no filter), 밉맵 끔, 무압축, Wrap Clamp | 보간 흐림·밉맵 전환·가장자리 반복을 배제하고 원본 픽셀을 유지 |
| 피벗 | 기본 Custom `(0.5, 0.25)` | 쥐처럼 지면 접촉점이 캔버스 하단 1/4에 맞춰진 원본의 공통 접지 기준 |
| 렌더 일관성 | 같은 투영·카메라·조명·팔레트·알파 여백·방향 순서 | 프레임 교체가 크기·명암·방향의 팝으로 보이지 않게 함 |

- `128px / PPU64`는 캔버스의 월드 폭을 보존하는 값이다. 실제 실루엣의 화면 크기까지 같게 하려면 Blender 렌더 카메라, 대상의 원점, 지면 높이, 알파 여백도 기존 기준과 같아야 하며 Unity 실제 화면에서 다시 대조한다.
- 정지 프레임도 같은 128px 세트의 방향별 첫 프레임을 사용한다. 정지용 별도 저해상도 이미지나 다른 Import 규격을 섞지 않는다.
- 저해상도 내부 렌더 또는 비정수 화면 확대가 별도로 적용되면 128px 원본만으로 흐림이 해결되지는 않는다. Point Import와 목표 출력 해상도·정수 배율의 실제 화면 검증을 함께 기록한다.

#### 예외 처리

- 피벗 `(0.5, 0.25)`은 같은 캔버스 구성과 지면 접촉 위치를 가진 원본에만 그대로 쓴다. 비행체, 부유물, 매우 긴 소품처럼 접지점이 다른 대상은 실제 접촉점에 맞춘 Custom Pivot을 정하고, 이유와 값을 작업 패킷에 기록한다.
- 화면 점유율 또는 프레임 여백 때문에 128px 캔버스가 부족한 대상은 임의로 PPU만 바꾸지 않는다. 더 큰 캔버스를 쓸 때는 같은 월드 폭을 유지하는 정수쌍(예: `256px / PPU128`)을 비교하고, 사용자 승인·QA 기록을 거친다.
- UI, 픽셀 폰트, 3D 환경의 원거리 바닥·벽은 이 기준의 대상이 아니다. 각 대상은 이 가이드의 UI·환경 텍스처 규칙과 장면별 밉맵 비교를 따른다.

#### QA 수용 기준

- 모든 방향·프레임이 실제 렌더된 `128×128` RGBA PNG이며, 단순 확대 파일이 아님을 원본 렌더 설정과 산출물 크기로 확인한다.
- Unity Import가 Sprite Single, PPU64, 해당 피벗, Point, 밉맵 끔, 무압축, Clamp인지 전체 프레임에서 확인한다.
- 기존 기준과 캔버스 월드 폭이 모두 2 Unity unit이고, 이동 루트·콜라이더·접지·상호작용 위치가 바뀌지 않는지 확인한다.
- 정지·이동·대각선·빠른 8방향 전환에서 동일 v4 세트만 표시되고, 선명도·크기·피벗·명암이 바뀌는 팝이나 외곽 흔들림이 없는지 실제 Play 화면에서 확인한다.
- 목표 출력 해상도에서 정지·연속 WASD 이동을 확인해 흐림·밉맵 전환·서브픽셀 떨림이 없고, 콘솔 오류·경고가 없는 것을 QA 기록과 사용자 시각 확인으로 남긴다.

### 해상도·팔레트·UV

- 환경 시작 텍스처는 작은 격자를 우선한다. 소품은 `16×16` 또는 `32×32`, 큰 표면은 `32×32` 또는 `64×64`에서 먼저 비교한다. 캐릭터 스프라이트 셀 크기는 화면 점유율과 방향별 꼬리·돌기 여백을 시험한 뒤 별도로 결정한다.
- 장면은 제한된 공용 팔레트를 두고, 한 대상은 보통 3~5개 명도 단계로 큰 형태를 읽게 한다. 위험·상호작용·보상 색상은 배경 팔레트와 겹치지 않게 예약한다.
- UV 섬은 텍셀 격자에 맞춰 배치하고, 한 면 안에서 불필요한 비정수 스케일과 회전을 줄인다. 아틀라스 가장자리에는 충분한 여백을 둬 인접 색이 새지 않게 한다.
- 손으로 그린 픽셀 단위의 계단 윤곽과, 모델 윤곽이 낮은 해상도에서 만드는 계단은 서로 충돌할 수 있다. 중요한 외곽은 텍스처 장식보다 모델 실루엣으로 우선 해결한다.

### 권장 Import 출발점

| 용도 | 필터링 | 밉맵 | 확인할 점 |
| --- | --- | --- | --- |
| HUD, 아이콘, 픽셀 폰트 | Point (no filter) | 끔 | 정수 배율과 픽셀 정렬을 지키고, UI가 흐려지지 않는지 확인 |
| 8방향 캐릭터 스프라이트 | Point (no filter) | 끔으로 시작 | 피벗·알파 여백·월드 크기·방향 전환 중 외곽 흔들림을 확인 |
| 고정 카메라의 근거리 3D 소품 | Point (no filter) | 끔으로 시작 | 이동·카메라 추적 중 픽셀 가장자리가 흔들리지 않는지 확인 |
| 멀리 보이거나 비스듬한 바닥·벽 | Point와 밉맵 유무를 비교 검증 | 필요 시 켬 | 밉맵을 끄면 반짝임이 생길 수 있으므로, 의도된 선명도와 안정성을 장면별로 비교 |

압축 방식, 최대 크기, 플랫폼별 Override는 실제 텍스처와 목표 PC 해상도를 기준으로 검증한 뒤 정한다. 모든 텍스처에 하나의 Import 규칙을 강제하지 말고, UI·캐릭터·원거리 환경의 화면 문제를 분리해 기록한다.

## 5. URP 저해상도 렌더링과 픽셀화 적용 순서

### 선택지

1. **낮은 내부 해상도 + 정수 배율 확대를 먼저 권장한다.** 고정 저해상도 Render Texture로 씬을 렌더한 뒤, 화면에는 2배·3배·4배처럼 정수 배율로만 확대한다. 비정수 배율이 필요하면 화면을 흐리게 늘리기보다 레터박스 또는 필러박스로 남는 영역을 처리한다.
2. **픽셀화 후처리는 비교안으로 사용한다.** 목표 화면 비율 또는 UI 분리 때문에 고정 Render Texture만으로 원하는 결과를 얻지 못할 때, URP Full Screen Pass·Renderer Feature·셰이더 기반 격자화를 별도 승인 작업으로 시험한다. 후처리가 전체 화면인지 3D 월드만인지도 명시한다.

### 시작 해상도 후보

| 내부 기준 해상도 | 화면 비율 | 정수 배율이 맞는 대표 출력 | 용도 |
| --- | --- | --- | --- |
| `320×180` | 16:9 | `1280×720` 4배, `1600×900` 5배, `1920×1080` 6배 | 첫 프로토타입 비교의 기본 후보 |
| `480×270` | 16:9 | `1920×1080` 4배 | UI·모델 세부가 너무 뭉개질 때의 비교 후보 |
| `640×360` | 16:9 | `1280×720` 2배, `1920×1080` 3배 | 더 큰 화면 정보가 필요한 경우의 비교 후보 |

이 후보는 PC 우선 프로토타입에서 화면 읽기와 성능을 비교하기 위한 기준이다. 실제 출력 해상도, Display 처리, Render Texture, URP Render Scale 값은 아직 확정하지 않는다.

### 적용 및 검증 순서

1. 목표 출력 해상도와 내부 해상도 한 쌍을 고르고, 정수 배율이 아닌 출력에서는 레터박스·필러박스 처리 방침을 정한다.
2. 고정 카메라와 대표 씬에서 정지 화면을 먼저 확인한다. 텍스처, 모델 외곽, HUD의 한 픽셀이 흐려지지 않아야 한다.
3. 쥐 이동, 대각선 이동, 카메라 추적, 배관·바닥 원근 이동을 녹화 또는 연속 관찰한다. 깜빡임·모아레·서브픽셀 떨림을 기록한다.
4. 낮은 내부 해상도만으로 문제가 해결되지 않을 때에만 픽셀화 후처리를 별도 비교한다. 후처리 적용 순서, UI 포함 여부, 성능, 콘솔 경고를 함께 검증한다.

동적 해상도, 시간 누적형 안티앨리어싱, 화면을 매 프레임 부드럽게 재샘플하는 효과는 픽셀 안정성을 해칠 수 있다. 초기 비교에서는 이 요소를 분리해 끈 상태와 켠 상태를 명확히 기록하고, 특정 기능을 실제로 유지할지는 시각 검증 뒤 결정한다.

## 6. 카메라와 픽셀 안정성

### 숙주 탐험: 쿼터뷰

- 쥐 숙주 모드는 고정 쿼터뷰를 우선한다. 시작 비교값으로는 수평 회전 약 `45°`, 하향 각도 약 `35~45°`를 두고, 원근이 필요하면 좁은 시야각(`28~35°`)을 비교한다.
- 캐릭터 원본 렌더 카메라는 런타임 카메라의 투영·하향 시점·화면 점유율과 맞춘다. 프리렌더드 방향과 환경 원근이 어긋나지 않도록 프로토타입에서는 카메라 회전과 연속 줌을 기본 지원하지 않는다.
- 픽셀 안정성이 최우선이면 직교(Orthographic) 카메라부터 확인한다. 원근 카메라를 쓸 경우에도 FOV, 카메라 높이, 추적 거리를 플레이 중 연속적으로 바꾸지 않는다.
- 카메라 추적의 감쇠, 아주 작은 회전, 비정수 위치 변화는 낮은 내부 해상도에서 윤곽을 흔들 수 있다. 추적 속도와 회전은 최소화하고, 문제가 보이면 카메라·대상의 화면상 이동을 픽셀 격자에 맞추는 방식을 별도 기술 검토한다.

### 내부 바이러스 미니게임: 탑다운 또는 고정 쿼터뷰

- 기본 비교안은 직교 탑다운이다. 움직이는 아레나가 아니라 고정 프레이밍으로 바이러스, 백혈구, 변이 조각의 충돌과 회피를 읽게 한다.
- 깊이를 보여줘야 할 때만 약한 기울기의 고정 쿼터뷰를 비교한다. 플레이어 이동 방향과 화면 방향의 대응이 숙주 모드와 크게 달라지지 않는지 확인한다.

### 공통 확인

- 카메라 흔들림, 줌, FOV 변화, 모션 블러는 처음부터 넣지 않는다. 피드백 필요성이 검증된 경우에만 낮은 해상도에서 읽히는 강도로 추가한다.
- 화면 가장자리에서 캐릭터 외곽이 잘리거나 1픽셀 단위로 진동하지 않는지, 대각선·긴 배관·난간처럼 고대비 선이 이동할 때 튀지 않는지 확인한다.

## 7. 조명, 그림자, 포그, 색보정

- 프로토타입은 한 개의 주 방향광과 단순한 환경광에서 시작한다. 형태를 읽기 위한 밝음·중간·어두움의 큰 덩어리를 만들고, 작은 광원 다수를 먼저 쌓지 않는다.
- 캐릭터 원본 렌더 조명은 모든 방향·프레임에서 동일하게 유지한다. 구운 명암과 Unity 실시간 조명이 중복되거나 3D 환경의 주광 방향과 반대로 읽히지 않게 한다.
- 접지감은 별도 단순 바닥 그림자 또는 승인된 투영 방식으로 처리하며, 원본 렌더에 포함된 그림자와 중복하지 않는다.
- 그림자는 하드 섀도 또는 제한된 부드러움 중 하나를 장면별로 비교한다. 그림자 거리·해상도·캐스케이드가 화면을 흐리게 하거나 깜빡이게 하면 범위를 줄이거나 끈 비교안을 함께 본다.
- 포그는 거리층과 하수도 분위기를 보조할 때만 사용한다. 배경과 위험 요소의 명도 대비를 지워서는 안 되며, 긴 그라데이션보다 제한 팔레트 안의 깊이 구분을 우선한다.
- 색보정은 공용 팔레트를 정돈하는 용도다. 과도한 Bloom, 색수차, 화면 노이즈, 강한 비네트는 픽셀 표면과 HUD 가독성을 망칠 수 있으므로 초기 기준에서는 제외한다.
- 초반 하수도 구간은 귀엽고 자연적인 위험을 읽게 하고, 장기적으로 인간 공간이 어두워지는 분위기는 이후 콘텐츠 범위에서 별도 기획한다. 이 가이드는 후반 콘텐츠를 추가하지 않는다.

## 8. UI와 텍스트

- HUD는 3D 월드 렌더와 분리해 선명하게 읽히게 한다. 전체 화면을 픽셀화할 때에도 UI까지 낮은 내부 해상도로 보낼지, 원래 출력 해상도에서 픽셀 정렬만 할지 먼저 결정하고 기록한다.
- 아이콘, 패널 테두리, 게이지, 픽셀 폰트는 정수 배율과 정수 좌표 정렬을 우선한다. 일반 벡터 폰트나 SDF 텍스트를 그대로 축소해 흐리게 보이게 하지 않는다.
- 현재 프로토타입 UI는 숙주 생명력, 면역 경계도, 현재 모드, 변이 조각 수, 변이 선택에 필요한 정보만 우선한다. 새 게이지나 장기 콘텐츠 UI를 이 문서만으로 추가하지 않는다.
- 경고·상호작용·성공·실패 피드백은 배경 팔레트와 다른 명도 또는 예약 색상으로 구분하고, 색만으로 의미를 전달하지 않도록 짧은 아이콘·텍스트·형태 변화도 함께 사용한다.

## 9. 플레이스홀더와 최종 아트의 구분

| 구분 | 프로토타입에서 허용되는 기준 | 최종 아트에서 별도 결정할 항목 |
| --- | --- | --- |
| 캐릭터 | 현재 3D 프리미티브 또는 임시 저폴리 원본에서 출력한 제한 팔레트 8방향 스프라이트로 게임플레이·파이프라인 검증 | 최종 원본 모델·리그, 애니메이션, 방향별 렌더, 아틀라스·압축 |
| 환경 모델 | 프리미티브·단순 저폴리 블록으로 크기, 충돌, 실루엣, 이동 가독성 검증 | 세부 소품, 최종 모듈, LOD |
| 환경 텍스처 | 제한 색상 블록·임시 픽셀 패턴으로 대상 구분 | 최종 팔레트, 텍스처 아틀라스, 손그림 디테일, 플랫폼별 압축 |
| 렌더링 | 한 개의 낮은 내부 해상도 비교안과 단순 조명으로 픽셀 안정성 검증 | 실제 Render Texture·Renderer Feature·후처리 구조, 성능 예산 |
| UI | 핵심 루프 이해용 게이지·프롬프트의 선명도 검증 | 최종 폰트, 아이콘 체계, 애니메이션, 접근성 세부 정책 |

회색 기본 프리미티브만 남기면 의도된 저폴리 스타일과 미완성 상태가 구분되지 않는다. 플레이스홀더라도 역할별 색상, 큰 실루엣, 최소한의 명도 규칙을 적용하고 문서·씬 이름에서 임시임을 표시한다.

## 10. AI 보조 래스터 제작 규칙

ChatGPT 등 이미지 생성 도구는 도트풍 3D의 **래스터 소스 초안**을 빠르게 비교하는 보조 수단이다. 생성 결과가 곧바로 게임 에셋이나 3D 제작 결과가 되는 것은 아니며, 아래 경계를 유지한다.

### 허용되는 보조 용도

- 쥐·바이러스·백혈구·하수도 환경의 콘셉트 시트와 제한 팔레트 비교안
- 캐릭터 3D 원본의 정면·측면 실루엣과 표면 색상 참고안
- 저해상도 픽셀 텍스처 초안과 색상 블록 연구
- 반복 가능성을 사람이 확인할 전제의 심리스 타일·배관 패턴 초안
- HUD 아이콘, 게이지 장식, 상태 표식의 초안

### 대체할 수 없는 작업

- 최종 저폴리 원본의 형태·폴리곤 밀도·실루엣 결정
- UV 전개, 리깅, 애니메이션, 8방향 렌더 설정, 공통 피벗·아틀라스 여백, 콜라이더
- Unity Import, 머티리얼, URP Renderer·후처리, 카메라·조명 적용
- 실제 플레이 중 가독성·픽셀 안정성·성능 검증

### 승인과 선별

- 실제 생성은 자산 묶음(예: 쥐 표면, 하수도 패턴, 내부 미니게임 아이콘)마다 사용자가 범위·용도·생성 수를 승인한 뒤에만 시작한다. 한 묶음의 승인은 다른 묶음이나 Unity 적용 승인이 아니다.
- 생성 전에는 대상, 화면 크기, 팔레트, 금지 요소, 카메라 기준, 반복 타일 여부를 적은 스타일 브리프와 자산 목록을 작업 패킷에 둔다.
- 생성 후에는 사람이 후보를 선별하고, 필요한 경우 손 편집·심리스 연결·팔레트 정리·투명 영역 정리를 거친다. 생성 결과를 무검토로 최종 텍스처나 UI에 넣지 않는다.
- 작업 패킷에는 프롬프트, 사용한 참조의 출처, 생성 도구·날짜, 후보/선택 변형, 선택 이유, 생성 원본과 최종 파일 위치를 남긴다. 공용 팔레트·실루엣·아이콘 규칙을 한 장의 일관성 시트로 함께 관리한다.

### 출처·권리·개인정보 경계

- 제3자 상표, 캐릭터, 로고, 특정 작가의 고유 화풍을 모사하도록 요청하거나 결과를 사용하지 않는다.
- 비공개 기획 자료, 개인 정보, 외부 계약 자료처럼 공개 권한이 없는 자료는 생성 도구 입력에 넣지 않는다.
- 도구의 이용 약관·상업 이용 조건·출력의 출처 요구를 자산 묶음 승인 때 확인하고, 결과가 기존 저작물과 혼동될 위험이 있으면 사용하지 않는다.
- 실제 생성 원본과 편집본의 보관 위치·라이선스 메모는 해당 자산 작업 패킷에 연결한다. 이 문서는 실제 파일 생성이나 권리 보증을 대신하지 않는다.

AI 보조 작업의 단계별 실행·역할·수용 기준은 `../../prototype/plans/rat-host-ai-assisted-art-workflow.md`를 따른다. AI로 방향별 프레임을 각각 생성해 형태가 흔들리는 결과를 최종 기준으로 사용하지 않으며, 이 규칙은 순수 2D 게임 전환이나 승인 없는 새 콘텐츠 추가를 허용하지 않는다.

## 11. 씬 검토 체크리스트

### 3D 원본, 스프라이트와 게임플레이

- [ ] 쥐, 바이러스, 백혈구의 3D 원본과 8방향 스프라이트가 실루엣·색·명도·비율을 일관되게 유지한다.
- [ ] 모든 방향·프레임의 공통 피벗·크기·여백이 맞고, 정지·대각선·빠른 전환에서 방향 팝핑·프레임 왕복·발 미끄러짐이 없다.
- [ ] 스프라이트 시각 자식이 3D 게임플레이 루트의 충돌·이동·위험 판정·상호작용을 바꾸지 않는다.
- [ ] 걷는 경로, 막힌 경로, 위험한 표면, 상호작용 지점이 장식 없이도 읽힌다.
- [ ] 작은 장식·얇은 면·고밀도 베벨이 낮은 해상도에서 깜빡이거나 역할을 흐리지 않는다.

### 픽셀 텍스처와 렌더링

- [ ] 캐릭터·소품·UI의 텍셀 밀도와 팔레트가 장면 안에서 과도하게 섞이지 않는다.
- [ ] Point 필터링과 밉맵 설정이 대상별로 기록되어 있고, 이동 중 흐림·반짝임을 확인했다.
- [ ] 내부 해상도와 출력 해상도의 배율이 정수인지, 비정수라면 레터박스·필러박스 방침이 명시되어 있다.
- [ ] 낮은 내부 해상도와 픽셀화 후처리 중 실제 적용안과 비교 결과가 기록되어 있다.

### 카메라·조명·UI

- [ ] 쿼터뷰 또는 탑다운 카메라가 플레이어 입력 방향과 충돌하지 않고, 이동·추적 중 서브픽셀 떨림이 없다.
- [ ] 조명, 그림자, 포그가 위험 요소와 HUD의 명도 대비를 지우지 않는다.
- [ ] 캐릭터의 구운 명암·바닥 그림자·3D 환경 주광이 충돌하지 않고, 벽·배관·소품과의 앞뒤 가림과 깊이 정렬이 맞는다.
- [ ] HUD·텍스트가 정수 배율에서 선명하고, 배경과 겹쳐도 핵심 게이지·목표를 읽을 수 있다.

### 승인과 검증

- [ ] 실제 모델·텍스처·Import·URP Renderer·카메라·후처리 변경은 별도 사용자 승인 범위와 작업 패킷에 기록했다.
- [ ] Unity 컴파일·콘솔·Play 체크와 목표 출력 해상도별 시각 검증 결과를 QA 기록에 남겼다.
- [ ] 쥐 숙주 프로토타입의 승인 범위를 넘어서는 새 콘텐츠·UI·시스템을 추가하지 않았다.
