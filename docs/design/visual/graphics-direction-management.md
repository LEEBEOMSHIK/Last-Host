# 그래픽 방향 관리 기준

## 목적과 적용 범위

이 문서는 `마지막 숙주 / The Last Host`의 그래픽 방향을 작업 간에 일관되게 판단하기 위한 영구 관리 기준이다. 캐릭터·환경·UI의 신규 시안, Blender 원본, 프리렌더드 스프라이트, Unity 저해상도 출력과 시각 QA는 이 문서를 먼저 확인한다.

- 게임플레이 공간·충돌·카메라는 Unity 3D로 유지한다.
- 저폴리 3D 캐릭터 원본을 8방향 프리렌더드 스프라이트로 표시하는 도트풍 하이브리드 2.5D 방향을 유지한다.
- 이 문서는 그래픽 판단 기준이며, 쥐 숙주 프로토타입 범위나 게임플레이 기능을 변경하지 않는다.
- 실제 Unity 설정값과 에셋 적용은 각 작업 패킷의 사용자 승인과 검증 기록으로 확정한다.

`pixel-lowpoly-3d-production-guide.md`는 제작 기술 가이드이고, 본 문서는 무엇을 만들고 어떤 화면을 통과로 볼지 정하는 관리 기준이다. 둘이 충돌하면 승인된 프로토타입 범위와 사용자 최신 결정이 우선한다.

## 참고 게임: 그래픽 판단용 벤치마크

사용자가 지정한 아래 게임은 **그래픽 언어를 판단하기 위한 참고**다. 게임 구조, 콘텐츠, 캐릭터, 스토리, UI 동작의 참고 대상으로 확대하지 않는다.

| 참고 게임 | 가져올 그래픽 판단 요소 | `마지막 숙주` 적용 방향 |
| --- | --- | --- |
| `Dave the Diver` | 작은 화면에서도 읽히는 조밀한 픽셀 군집, 명확한 깊이·명도 단계, 제한된 색 안에서의 풍부한 표면 정보 | 무작위 노이즈나 과도한 단색화를 피하고, 의도적인 픽셀 군집과 선택된 중간 명도색으로 쥐·환경의 부피와 재질을 읽힌다. |
| `The Binding of Isaac` | 즉시 구분되는 실루엣, 강한 명도·색 대비, 위험물·상호작용물의 우선순위가 분명한 화면 | 플레이어·위험·수집물은 배경보다 먼저 읽혀야 하며, 화면이 작아도 이동 방향과 행동 상태를 구분할 수 있어야 한다. |

참고 게임의 캐릭터, 적, 방, 아이템, 아이콘, UI, 배치, 색 조합, 화면 구성, 고유 모티프 또는 자산을 복제하지 않는다. 비교는 "픽셀 밀도·실루엣·가독성·명도 깊이" 수준의 원칙에 한정한다.

## 프로젝트 그래픽 원칙

1. **3D 기반을 유지한다.** 환경과 공간감은 3D로, 캐릭터는 동일한 저폴리 3D 원본에서 만든 8방향 스프라이트로 표현한다. 순수 2D 복셀·스프라이트 게임으로 바꾸지 않는다.
2. **의도적인 픽셀 밀도.** 윤곽과 표면은 큼직한 단색 덩어리만으로 끝내지 않는다. 읽힘을 해치지 않는 범위에서 불투명 중간 명도색과 군집을 써서 재질·부피·접지를 표현한다. 디더링·반투명·고립 픽셀은 효과가 분명할 때만 쓴다.
3. **즉시 읽히는 우선순위.** 쥐의 방향·이동·정지, 위험, 수집물, 상호작용 지점이 배경과 구분되어야 한다. 작은 화면·대각 이동·정지 화면에서 모두 확인한다.
4. **안정된 저해상도 출력.** 픽셀 아트처럼 보이려면 보간 흐림, 서브픽셀 떨림, 이동·정지 시 외형 변화가 없어야 한다. 스프라이트, 카메라, 출력 해상도는 함께 검증한다.
5. **월드와 UI의 역할 분리.** 저해상도 월드 출력은 허용하되, HUD·텍스트는 별도 레이어에서 선명하게 유지한다. UI는 참고 게임의 고유 화면을 모사하지 않는다.

## 현재 기준과 시험안 경계

| 구분 | 상태 | 적용 범위 |
| --- | --- | --- |
| 기본 공통 표시 기준 | 유지 | 쥐 걷기 스프라이트는 `128×128`, `PPU 64`, Point, Mipmap Off, Uncompressed, Clamp을 현재 비교 기준으로 사용한다. |
| v5b 픽셀 처리 시험안 | 사용자 시각 수용 대기 | 공통 팔레트(32-entry, 실제 27색), 이진 알파, 무디더, `960×540` Point 월드 출력, 시각·쿼터뷰 픽셀 스냅은 쥐 시험안에만 적용됐다. 사용자 승인 전에는 다른 오브젝트의 공통 규격으로 확대하지 않는다. |
| 참고 게임 반영 | 확정 | 위 참고 게임의 그래픽 판단 원칙은 이후 시안·검토에 적용한다. 단, 개별 해상도·팔레트 수·렌더 설정을 자동으로 확정하지 않는다. |

## 작업별 적용 규칙

### 캐릭터와 게임플레이 오브젝트

- 신규 캐릭터·바이러스·백혈구·수집물은 먼저 실루엣, 방향, 상태 변화, 배경 대비를 검토한다.
- 캐릭터 원본과 방향별 스프라이트는 동일한 외형 기준을 유지하며, 이동 중 프레임 차이가 의도하지 않은 크기·선명도 변화로 보이지 않아야 한다.
- 팔레트 수나 알파 규칙은 이 문서만으로 고정하지 않는다. 시험안에서 통과한 값은 사용자 수용 뒤에만 공통 규격으로 승격한다.

### 3D 환경

- 환경은 3D 깊이와 충돌·가림을 유지하되, 바닥·벽·소품의 표면 밀도와 명도 단계를 캐릭터 가독성을 해치지 않게 조절한다.
- 과도한 텍스처 노이즈, 보간 흐림, 너무 미세한 디테일로 플레이어·위험 식별을 늦추지 않는다.
- 환경에 새 저해상도 렌더 또는 포스트 처리 규칙을 적용할 때는 캐릭터·HUD와 함께 실제 플레이 화면을 검증한다.

### UI

- 월드 출력과 분리된 선명한 레이어를 유지한다.
- 정보 위계는 분명히 하되, 참고 게임의 아이콘·프레임·배치·색 체계를 복제하지 않는다.

## 시각 수용 검토표

신규 그래픽 시안이나 렌더 변경을 완료로 주장하기 전에 아래를 확인한다.

- [ ] 3D 공간과 프리렌더드 8방향 캐릭터라는 하이브리드 2.5D 구조를 유지하는가?
- [ ] 정지·이동·대각 이동·방향 전환에서 크기, 선명도, 윤곽이 의도치 않게 변하지 않는가?
- [ ] 보간 흐림, 서브픽셀 떨림, 프레임 간 팔레트 흔들림이 없는가?
- [ ] 플레이어·위험·수집물·상호작용 지점이 배경보다 먼저 읽히는가?
- [ ] 화면을 작게 보아도 실루엣과 명도 단계가 살아 있는가?
- [ ] 픽셀 밀도와 표면 정보가 단순한 저해상도 확대가 아니라 의도된 도트 표현으로 보이는가?
- [ ] 참고 게임의 고유 자산·구성·시각 정체성을 복제하지 않았는가?
- [ ] 실제 Unity Play 화면과 독립 QA 기록으로 확인했는가?

## 변경 관리

- 참고 게임의 추가·교체, 공통 해상도·팔레트·출력 규격의 승격은 사용자 결정과 작업 패킷 기록이 필요하다.
- 그래픽 시험안은 명칭, 적용 대상, 검증 결과, 남은 위험, 사용자 수용 상태를 `_workspace/active/<작업ID>/`에 기록한다.
- 사용자 수용 전 시험안은 기존 공통 기준을 대체하지 않는다.
