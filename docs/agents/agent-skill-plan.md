# 에이전트/스킬 구성 계획

최종 수정일: 2026-07-01

## 목적

이 문서는 `마지막 숙주 / The Last Host`를 Codex 중심으로 개발하기 위해 필요한 에이전트 역할, 스킬, 작업 배정 절차, 산출물 수신 절차, 승인 게이트를 정리한다. 현재 프로젝트 종속 에이전트는 `.agents/`에, 프로젝트 종속 Codex 스킬은 `.codex/skills/`에 둔다.

## 기본 원칙

- 모든 설명과 프로젝트 문서는 한국어로 작성한다.
- 에이전트는 독립적으로 판단하되, 프로젝트 방향은 `AGENTS.md`와 `docs/` 문서를 따른다.
- 작업별 세부 참조는 `docs/agents/agent-reference-map.md`를 따른다.
- 에이전트에게 일을 맡길 때는 목적, 입력, 산출물, 금지 범위, 승인 필요 항목을 명확히 준다.
- 에이전트 결과물은 자동 반영하지 않는다. 조정자가 검토하고 필요한 경우 사용자 승인을 받은 뒤 반영한다.
- 프로젝트 방향, 범위, 승인 게이트에 영향을 주는 주요 산출물은 사용자 최종 승인 전에 프로젝트 총괄 관리자 에이전트가 내부 승인 여부를 검토한다.
- 구현, Unity 프로젝트 생성, 패키지 추가, 스킬 생성은 별도 승인 후 진행한다.
- 스킬 사용은 에이전트 위임을 대체하지 않는다. QA/검증 에이전트나 총괄 관리자 에이전트가 일한 것으로 보고하려면 해당 역할의 산출물과 판정이 있어야 한다.

## 프로젝트 총괄 승인 구조

프로젝트 총괄 관리자 에이전트는 사용자 최종 승인 전 내부 승인자다. 최종 결정권은 사용자에게 남기고, 에이전트 산출물이 사용자에게 올라가기 전에 승인 가능 상태인지 판단한다.

역할 구분:

- 프로젝트 총괄 관리자 에이전트: 방향, 범위, 승인 게이트, 산출물 충돌 여부를 판정한다.
- 프로젝트 조정 에이전트: 작업을 분해하고 담당 에이전트를 배정하며 결과를 통합한다.
- 전문 에이전트: 기획, Unity 구조, 게임플레이, 비주얼, 구현, 검증, 문서 등 각 전문 영역의 산출물을 만든다.
- 구현 에이전트: 승인된 범위 안에서 실제 코드, 테스트, 씬, 프리팹, 설정 변경을 수행한다.

총괄 관리자 판정은 다음 네 가지 중 하나로 남긴다.

- `내부 승인 가능`: 사용자에게 최종 확인을 올릴 수 있다.
- `수정 필요`: 담당 에이전트나 조정자가 보완한 뒤 다시 검토한다.
- `사용자 결정 필요`: 방향, 범위, 비용, 승인 게이트 판단이 필요해 사용자에게 질문을 올린다.
- `보류`: 외부 자료, 실행 결과, 사용자 입력이 없어 판단할 수 없다.

총괄 관리자 에이전트는 최종 사용자 승인, Unity 프로젝트 수정, 코드 작성, 패키지 추가, 에셋 생성을 직접 실행하지 않는다.

## 루프 엔지니어링 운영 원칙

프로젝트 작업은 루프 엔지니어링 방식으로 운영한다. 한 번의 루프는 사용자 목표를 작업 상태로 만들고, 에이전트에게 배정하고, 산출물을 검토하고, 검증한 뒤, 사용자에게 필요한 확인 사항만 보고하고, 다음 작업 후보를 남기는 단위다.

기본 루프:

1. 사용자 목표를 접수한다.
2. 프로젝트 총괄 관리자 에이전트가 방향과 승인 게이트를 1차 확인한다.
3. 프로젝트 조정 에이전트가 `_workspace/active/<작업ID>/` 작업 패킷을 만든다.
4. 담당 에이전트가 산출물을 만든다.
5. 조정자가 산출물, 충돌, 누락, 승인 필요 항목을 통합한다.
6. 총괄 관리자 에이전트가 내부 승인 가능 여부와 문제 사안을 판정한다.
7. 사용자에게 확인할 파일, 핵심 확인 사항, 결정 필요 항목만 보고한다.
8. 승인된 내용만 문서, 코드, 씬, 에셋, 커밋에 반영한다.
9. 완료 기록과 검증 기록을 `_workspace/completed/`에 보관한다.
10. 다음 작업 후보를 남기거나 다음 루프로 넘긴다.

완료 전 차단 게이트:

- 코드, 씬, ProjectSettings, 승인 문서, 운영 문서 변경은 완료 또는 커밋 전에 `docs/agents/loop-engineering-gates.md`를 따른다.
- `_workspace` 작업 패킷, 담당 산출물, QA/검증 에이전트 기록, 총괄 관리자 판정이 모두 있어야 완료로 보고한다.
- 메인 에이전트가 직접 검증 명령을 실행한 것만으로는 QA/검증 에이전트 검증으로 보지 않는다.
- 메인 에이전트는 기본적으로 조정자/통합자이며, 코드/씬/테스트/ProjectSettings 변경은 구현 담당 에이전트에 배정한다.
- 메인 에이전트 직접 구현 예외는 사용자 명시 승인과 `agent-activity.md`의 예외 사유가 있을 때만 허용한다.
- 총괄 관리자 판정이 `내부 승인 가능`이 아니면 커밋 완료나 작업 완료로 보고하지 않는다.

문제 사안 보고 기준:

- 사용자 승인 없이 진행할 수 없는 승인 게이트가 발견된 경우
- 프로토타입 범위, 게임 방향, Unity 방향이 기존 문서와 충돌하는 경우
- 에이전트 산출물이 서로 충돌하거나 책임 경계가 불명확한 경우
- 입력 자료가 부족해 추정으로 진행해야 하는 경우
- 검증 명령을 실행할 수 없거나 결과가 완료 판단을 뒷받침하지 못하는 경우
- `_workspace` 작업 패킷, 진행 로그, 완료 기록이 누락된 경우
- QA/검증 에이전트 기록 또는 총괄 관리자 판정이 누락된 경우
- 다음 루프로 넘기기 전에 사용자 결정이 필요한 경우

문제 사안은 사용자에게 장황한 작업 로그로 전달하지 않는다. `문제`, `영향`, `선택지`, `추천`, `확인할 파일`만 짧게 정리한다.

## 에이전트 위임 우선 규칙

담당 에이전트가 정의된 작업은 조정자가 직접 최종 산출물을 작성하지 않는다. 먼저 `_workspace/active/<작업ID>/`에 작업 패킷을 만들고, 담당 에이전트가 작업할 입력과 산출물 위치를 지정한다.

Codex 스킬을 사용해 조사, 구현, 검증을 도와도 이것은 담당 에이전트 산출물로 자동 인정하지 않는다. 담당 에이전트가 정의된 작업은 산출물에 담당 역할, 입력 자료, 검증 내용, 남은 위험을 명시해야 한다.

적용 대상:

- 구현 계획서
- Unity 구조안
- 핵심 루프 작업 분해
- 수용 기준과 테스트 시나리오
- 비주얼 기준
- 검증 체크리스트
- 문서/릴리즈 정리안
- 게임플레이 코드와 테스트 구현
- Unity 씬, 프리팹, 입력, 카메라, UI, 설정 통합 구현

조정자의 역할:

1. 작업 목적과 범위를 정한다.
2. 담당 에이전트와 보조 에이전트를 고른다.
3. 입력 자료, 산출물, 금지 범위, 승인 필요 항목을 작업 패킷에 쓴다.
4. 에이전트 산출물을 `_workspace/active/<작업ID>/artifacts/` 또는 `handoff.md`로 받는다.
5. 산출물을 검토한 뒤 사용자 승인 여부에 따라 `docs/`, 코드, 씬, 에셋에 반영한다.

예외:

- 사용자가 명시적으로 조정자에게 직접 문서 작성을 요청한 경우
- 담당 에이전트가 정의되어 있지 않은 작은 정리 작업
- 긴급한 작업 트리 정리처럼 산출물이 에이전트 전문 영역이 아닌 경우

예외에 해당하더라도 승인 게이트가 있는 작업은 사용자 승인을 먼저 받는다.

## 사용자 확인 보고 규칙

`_workspace`는 이력 보관과 에이전트 간 작업 배정을 위한 작업영역이다. 조정자는 최종 보고에서 `_workspace` 전체를 사용자 확인 목록으로 던지지 않고, 사용자가 직접 봐야 하는 파일만 선별해 전달한다.

보고할 때는 다음 형식을 기본으로 한다.

```text
확인할 파일:
- <파일 경로>: 핵심 확인 사항 1~3개

결정 필요:
- <사용자 판단이 필요한 항목>
```

파일별 핵심 확인 사항에는 다음 중 필요한 내용만 짧게 적는다.

- 이 파일을 봐야 하는 이유
- 사용자가 승인하거나 반려해야 하는 결정
- 프로토타입 범위, 승인 게이트, 구현 방향과 관련된 위험
- 다음 작업으로 넘어가기 전에 확인해야 하는 누락 항목

`task.md`, `work-log.md`, `handoff.md`, `verification.md` 같은 작업 기록은 근거 확인이 필요하거나 사용자가 요청한 경우에만 보조 링크로 제시한다.

## 전체 운영 구조

프로젝트 운영은 다음 구조를 기본으로 한다. 이 구조 전체가 루프 1회다.

1. 사용자가 목표를 제시한다.
2. 프로젝트 총괄 관리자 에이전트가 목표가 현재 방향과 승인 게이트에 충돌하는지 1차 확인한다.
3. 조정자가 목표를 작업 단위로 분해한다.
4. 조정자가 `_workspace/active/<작업ID>/` 작업 폴더를 만든다.
5. 조정자가 적절한 에이전트를 선택한다.
6. 각 에이전트가 지정된 범위 안에서 조사, 설계, 구현, 검증을 수행하고 작업 폴더에 기록한다.
7. 각 에이전트는 정해진 형식으로 결과를 반환한다.
8. 조정자가 충돌, 누락, 승인 필요 항목을 정리한다.
9. 프로젝트 총괄 관리자 에이전트가 사용자 최종 승인 전 내부 승인, 수정 필요, 사용자 결정 필요를 판정한다.
10. 사용자 승인 후 문서, 코드, 에셋, 커밋에 반영한다.
11. 완료 시 `_workspace/completed/<완료일>-<작업ID>/`에 완료 기록을 보관한다.

## 에이전트 목록

### 1. 프로젝트 총괄 관리자 에이전트

역할:

- 사용자 최종 승인 전 내부 승인자 역할을 맡는다.
- 프로젝트 전체 방향, 범위, 승인 게이트를 점검한다.
- 에이전트 산출물이 서로 충돌하거나 누락된 부분이 있는지 확인한다.
- 사용자에게 올릴 확인 파일과 핵심 확인 사항을 압축한다.
- 산출물을 `내부 승인 가능`, `수정 필요`, `사용자 결정 필요`, `보류`로 판정한다.

주요 입력:

- 사용자 요청
- 조정자 통합 보고
- 에이전트 산출물
- `AGENTS.md`
- `docs/agents/agent-skill-plan.md`

산출물:

- 내부 승인 판정
- 수정 필요 항목
- 사용자 결정 필요 항목
- 사용자 확인 파일 목록

### 2. 프로젝트 조정 에이전트

역할:

- 사용자 목표를 작업 단위로 나눈다.
- 담당 에이전트를 선택하고 작업을 배정한다.
- 에이전트 결과물을 통합한다.
- 승인 필요 항목을 사용자에게 올린다.
- 전체 문서와 구현 범위가 충돌하지 않는지 확인한다.

주요 입력:

- 사용자 요청
- `AGENTS.md`
- `docs/project/project-prep.md`
- `docs/agents/agent-skill-plan.md`

산출물:

- 작업 분해안
- 담당 에이전트 지정
- 승인 필요 항목 목록
- 최종 통합 보고

### 3. 기획 정리 에이전트

역할:

- 원본 `.docx`와 Markdown 문서의 차이를 관리한다.
- 기획 내용을 구현 가능한 단위로 정리한다.
- 미확정 항목과 승인 필요 항목을 분리한다.
- 범위가 커지는 요청을 감지한다.

주요 입력:

- `C:\project\game\last_host\last_host_game_plan.docx`
- `docs/design/game-design-summary.md`
- `docs/prototype/official/rat-host-prototype.md`

산출물:

- 기획 요약
- 범위 변경 제안
- 승인 질문
- 문서 갱신안

### 4. Unity 아키텍처 에이전트

역할:

- Unity 버전, URP, 폴더 구조, 씬 구조를 계획한다.
- 시스템 경계와 의존성을 설계한다.
- 구현 전 필요한 패키지와 설정을 정리한다.
- Unity 프로젝트 생성 전 승인 체크리스트를 만든다.

주요 입력:

- `docs/project/project-prep.md`
- `docs/unity/unity-mcp-setup.md`
- `docs/prototype/official/rat-host-prototype.md`
- Unity 관련 사용자 승인사항

산출물:

- Unity 프로젝트 구조안
- 씬 구성안
- 시스템 모듈 경계
- 패키지/설정 승인 목록

### 5. 게임플레이 루프 에이전트

역할:

- 쥐 숙주 조종, 면역 경계도, 내부 미니게임, 변이 선택의 핵심 루프를 설계한다.
- 기능을 작은 구현 단위로 나눈다.
- 구현 시 테스트 가능한 조건을 함께 제안한다.

주요 입력:

- `docs/prototype/official/rat-host-prototype.md`
- Unity 아키텍처 에이전트 산출물

산출물:

- 핵심 루프 작업 목록
- 기능별 입력/출력 정의
- 성공/실패 조건
- 테스트 시나리오

### 6. 비주얼/테크아트 에이전트

역할:

- 도트풍 3D 비주얼 규칙을 정리한다.
- 저폴리 모델, 픽셀 텍스처, 저해상도 렌더링 방향을 관리한다.
- 카메라, 조명, 후처리 기준을 제안한다.
- 플레이스홀더 에셋과 최종 에셋을 구분한다.

주요 입력:

- `docs/design/game-design-summary.md`
- `docs/project/project-prep.md`
- 사용자 비주얼 승인사항

산출물:

- 비주얼 스타일 가이드
- 카메라/렌더링 기준
- 플레이스홀더 에셋 목록
- 승인 필요한 아트 방향

### 7. 게임플레이 구현 에이전트

역할:

- 승인된 게임플레이 C# 코드와 관련 테스트를 실제로 구현한다.
- 상태 모델, 컨트롤러, 입력 처리, 게임 루프 로직을 기존 모듈 경계 안에서 수정한다.
- 구현한 변경의 테스트 가능 조건과 남은 위험을 기록한다.
- 씬 연결이나 직렬화 값이 필요하면 Unity 씬/통합 구현 에이전트로 넘긴다.

주요 입력:

- 작업 패킷
- 승인된 구현 계획
- 게임플레이 루프 에이전트 산출물
- 기존 코드와 테스트

산출물:

- C# 코드 변경
- EditMode 테스트 변경
- 구현 결과 핸드오프

### 8. Unity 씬/통합 구현 에이전트

역할:

- 승인된 씬, 프리팹, 입력, 카메라, UI, 빌드 설정 연결을 실제로 구현한다.
- Unity MCP를 사용한 씬 저장과 컴포넌트 연결 결과를 기록한다.
- 런타임 코드 변경이 필요하면 게임플레이 구현 에이전트로 넘긴다.

주요 입력:

- 작업 패킷
- Unity 아키텍처 에이전트 산출물
- 비주얼/테크아트 기준
- 구현된 컴포넌트

산출물:

- 씬/프리팹/설정 변경
- Unity 연결 결과
- Play 확인 기록

### 9. QA/검증 에이전트

역할:

- 완료 주장 전에 검증 절차를 정의한다.
- Unity 테스트, 빌드, 플레이 검증 체크리스트를 만든다.
- 기능별 수용 기준을 관리한다.
- 회귀 위험과 누락 테스트를 찾는다.

주요 입력:

- 구현 계획
- 변경 diff
- 테스트 로그
- 플레이 검증 결과

산출물:

- 검증 체크리스트
- 테스트 결과 요약
- 발견된 문제
- 완료 가능 여부 판단

### 10. 문서/릴리즈 에이전트

역할:

- 변경사항을 문서에 반영한다.
- 커밋 메시지, 변경 요약, 릴리즈 노트를 정리한다.
- 승인된 결정과 실제 구현이 어긋나지 않는지 확인한다.

주요 입력:

- 변경 diff
- 승인된 설계 문서
- 테스트 결과

산출물:

- 문서 수정안
- 커밋 요약
- 릴리즈/마일스톤 요약
- 남은 작업 목록

## 추가 스킬 생성 승인 절차

추가 Codex 스킬을 생성할 때는 다음 순서를 따른다.

1. 만들 스킬 이름을 정한다.
2. 스킬 생성 위치를 정한다.
   - 기본 후보: 사용자 Codex 스킬 폴더
   - 프로젝트 로컬 후보: 저장소 내부의 별도 스킬 자료 폴더
3. 스킬에 포함할 리소스를 정한다.
   - `SKILL.md`
   - `agents/openai.yaml`
   - `references/`
   - 필요한 경우 `scripts/`
   - 필요한 경우 `assets/`
4. 사용자에게 생성 계획을 제시한다.
5. 사용자가 승인하면 스킬을 생성한다.
6. 스킬 검증 명령을 실행한다.
7. 검증 결과와 사용 예시를 문서화한다.

스킬 생성 전 승인 질문 형식:

```text
다음 스킬을 생성해도 될까요?

- 스킬명:
- 생성 위치:
- 목적:
- 포함 리소스:
- 검증 방법:
- 프로젝트에 미치는 영향:
```

## 작업 배정 절차

작업을 에이전트에게 맡길 때는 다음 절차를 따른다.

1. 작업 목적을 한 문장으로 쓴다.
2. 작업이 기획, 아키텍처, 구현, 비주얼, 검증, 문서 중 어디에 속하는지 분류한다.
3. 작업 ID를 `YYYY-MM-DD-short-topic` 형식으로 정한다.
4. `_workspace/active/<작업ID>/` 폴더를 만들고 템플릿 파일을 배치한다.
5. 담당 에이전트를 1개 정한다.
6. 구현 작업이면 게임플레이 구현 에이전트 또는 Unity 씬/통합 구현 에이전트 중 실제 변경 담당을 명시한다.
7. 필요한 경우 보조 에이전트를 지정한다.
8. 입력 자료를 명시한다.
9. 산출물 형식을 지정한다.
10. 변경 금지 범위를 적는다.
11. 승인 필요 항목을 적는다.
12. 완료 기준을 적는다.
13. 에이전트 결과를 받아 조정자가 검토한다.

## `_workspace` 사용 절차

에이전트 간 작업 추적은 `_workspace`를 사용한다.

### 진행 중 작업 폴더

진행 중 작업은 다음 위치에 둔다.

```text
_workspace/active/<작업ID>/
```

필수 파일:

- `task.md`
- `work-log.md`
- `handoff.md`
- `artifacts/`

### 완료 작업 폴더

작업 완료 시 다음 위치에 별도 완료 폴더를 만든다.

```text
_workspace/completed/<완료일>-<작업ID>/
```

필수 파일:

- `task.md`
- `work-log.md`
- `completion-report.md`
- `verification.md`
- `artifacts/`

### 완료 처리 기준

작업은 다음 조건을 만족해야 완료로 기록한다.

1. 작업 배정 내용이 `task.md`에 남아 있다.
2. 주요 진행 과정이 `work-log.md`에 남아 있다.
3. 수행 내용과 변경 파일이 `completion-report.md`에 정리되어 있다.
4. 실행한 검증과 미검증 항목이 `verification.md`에 기록되어 있다.
5. 사용자 승인 필요 항목이 남아 있으면 완료가 아니라 `승인 대기`로 표시한다.

## 유기적 협업 절차

에이전트 간 작업은 순차 흐름과 병렬 흐름을 구분한다.

### 순차 흐름

한 에이전트의 산출물이 다음 에이전트의 입력이 되는 경우 순차로 진행한다.

예시:

1. 기획 정리 에이전트가 프로토타입 범위를 확정한다.
2. Unity 아키텍처 에이전트가 구조를 설계한다.
3. 게임플레이 루프 에이전트가 구현 작업을 쪼갠다.
4. QA/검증 에이전트가 수용 기준을 만든다.
5. 문서/릴리즈 에이전트가 변경사항을 정리한다.

### 병렬 흐름

서로 독립적인 판단이 필요한 경우 병렬로 진행할 수 있다.

예시:

- Unity 아키텍처 에이전트: 프로젝트 구조안 작성
- 비주얼/테크아트 에이전트: 도트풍 렌더링 기준 작성
- QA/검증 에이전트: 프로토타입 수용 기준 작성

병렬 작업 후에는 프로젝트 조정 에이전트가 충돌을 정리한다.

## 핸드오프 규칙

다음 조건이 생기면 다른 에이전트에게 넘긴다.

- 사용자 최종 승인 전에 주요 산출물의 내부 승인이 필요한 경우: 프로젝트 총괄 관리자 에이전트로 넘긴다.
- 기획 질문이 구현 판단을 막는 경우: 기획 정리 에이전트로 넘긴다.
- Unity 구조나 패키지 결정이 필요한 경우: Unity 아키텍처 에이전트로 넘긴다.
- 기능 동작과 플레이 감각이 핵심인 경우: 게임플레이 루프 에이전트로 넘긴다.
- C# 게임플레이 코드와 테스트 구현이 필요한 경우: 게임플레이 구현 에이전트로 넘긴다.
- Unity 씬, 프리팹, 입력, 카메라, UI, 설정 연결이 필요한 경우: Unity 씬/통합 구현 에이전트로 넘긴다.
- 렌더링, 카메라, 에셋 기준이 필요한 경우: 비주얼/테크아트 에이전트로 넘긴다.
- 완료 여부를 주장해야 하는 경우: QA/검증 에이전트로 넘긴다.
- 문서와 커밋 정리가 필요한 경우: 문서/릴리즈 에이전트로 넘긴다.

## 충돌 해결 절차

에이전트 결과가 충돌하면 다음 순서로 처리한다.

1. 원본 `.docx`와 `docs/` 문서를 확인한다.
2. `AGENTS.md`의 승인 게이트를 확인한다.
3. 프로젝트 총괄 관리자 에이전트가 내부 승인 가능, 수정 필요, 사용자 결정 필요를 판정한다.
4. 충돌이 단순 구현 방식이면 조정자가 보수적인 선택을 제안한다.
5. 게임 방향, 범위, 비용, 일정에 영향을 주면 사용자에게 승인 질문을 올린다.
6. 승인된 결정을 관련 문서에 반영한다.

## 승인 게이트

다음 항목은 에이전트가 제안할 수는 있지만, 사용자 승인 없이 실행하지 않는다.

- Unity 프로젝트 생성
- Unity 버전 확정
- URP 확정
- 패키지 설치
- Unity MCP 설정 변경 또는 MCP를 통한 프로젝트 변경
- 실제 Codex 스킬 생성
- 새 에이전트 역할 추가
- 기존 에이전트 책임 변경
- 게임플레이 코드 작성
- 에셋 생성
- 프로토타입 범위 확대
- 원본 기획과 다른 방향으로 문서 수정

## 작업 상태 정의

작업 상태는 다음 용어를 사용한다.

- `제안`: 아직 승인되지 않은 아이디어
- `승인 대기`: 사용자의 결정이 필요한 상태
- `승인됨`: 사용자가 진행을 허가한 상태
- `진행 중`: 에이전트가 작업 중인 상태
- `검토 중`: 산출물 검토 단계
- `수정 필요`: 결과물에 보완이 필요한 상태
- `완료`: 검증과 문서 반영이 끝난 상태
- `보류`: 외부 결정이나 자료가 필요해 멈춘 상태

## 운영 예시

### 예시 1: Unity 프로젝트 생성 준비

1. 사용자가 Unity 프로젝트 생성을 요청한다.
2. 프로젝트 총괄 관리자 에이전트가 승인 게이트에 걸리는 작업임을 확인한다.
3. 프로젝트 조정 에이전트가 작업을 접수한다.
4. Unity 아키텍처 에이전트가 Unity 버전, URP, 폴더 구조안을 작성한다.
5. 비주얼/테크아트 에이전트가 도트풍 3D 렌더링 요구를 확인한다.
6. QA/검증 에이전트가 프로젝트 생성 후 확인할 체크리스트를 만든다.
7. 프로젝트 총괄 관리자 에이전트가 사용자에게 올릴 승인 질문을 검토한다.
8. 조정자가 사용자에게 승인 질문을 올린다.
9. 승인 후 Unity 프로젝트를 생성한다.

### 예시 2: 쥐 숙주 이동 구현 준비

1. 게임플레이 루프 에이전트가 쥐 이동의 최소 기능을 정의한다.
2. Unity 아키텍처 에이전트가 입력, 카메라, 씬 의존성을 확인한다.
3. QA/검증 에이전트가 이동 수용 기준을 만든다.
4. 프로젝트 총괄 관리자 에이전트가 구현 계획을 사용자에게 올릴 수 있는지 판정한다.
5. 조정자가 구현 계획을 사용자에게 보고한다.
6. 승인 후 구현 작업을 시작한다.

### 예시 3: 도트풍 3D 비주얼 기준 정리

1. 비주얼/테크아트 에이전트가 렌더링 기준을 제안한다.
2. Unity 아키텍처 에이전트가 URP 설정 가능성을 확인한다.
3. QA/검증 에이전트가 화면 검증 기준을 만든다.
4. 프로젝트 총괄 관리자 에이전트가 비주얼 방향과 승인 필요 항목을 검토한다.
5. 조정자가 사용자에게 최종 방향 승인을 요청한다.

## 현재 생성 완료 항목

현재 프로젝트 안에 다음 구성이 생성되어 있다.

1. `.agents/agent-roster.md`와 역할별 에이전트 파일 10개
2. `.codex/skills/` 아래 프로젝트 로컬 스킬 5개
3. 각 스킬의 `SKILL.md`
4. 각 스킬의 `agents/openai.yaml`
5. 각 스킬의 `references/` 기준 문서 1개
6. `_workspace/` 에이전트 작업영역과 완료 추적 템플릿
7. `docs/agents/agent-reference-map.md` 작업별 참조 문서 색인
8. `.codex/config.toml` 프로젝트 로컬 Codex MCP 설정
9. `docs/unity/unity-mcp-setup.md` Unity MCP 설정과 운영 문서

## 다음 승인 필요 항목

다음 작업은 별도 사용자 승인이 필요하다.

1. 현재 프로젝트 로컬 스킬을 사용자 전역 Codex 스킬 폴더에 설치할지 여부
2. 추가 에이전트 또는 추가 스킬 생성, 제거, 책임 범위 변경 여부
3. 쥐 숙주 1차 프로토타입 범위를 넘어서는 새 숙주, 캠페인, 백신, 인간 단계 콘텐츠 추가 여부
4. 신규 패키지 설치 또는 새 아트 에셋 생성 여부
5. 루프 엔지니어링 완료 전 게이트를 예외적으로 생략해야 하는 경우
